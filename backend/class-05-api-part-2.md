# 2. Authentication vs Authorization

Very important difference.

| Term           | Meaning                     |
| -------------- | --------------------------- |
| Authentication | Who are you?                |
| Authorization  | What are you allowed to do? |

Example:

* Login → Authentication
* Admin dashboard access → Authorization

### Authorization Models

**RBAC** → Role Based Access Control

* admin
* user
* seller
* manager

**ABAC** → Attribute Based Access Control

Access based on:

* user
* time
* location
* action
* resource

Example:

> User can edit document only if he created it.

# 3. Stateful vs Stateless Systems

Very important backend concept.

## Stateful (Session Based Login)

Server stores session.

```text
Client → Login
Server → Create session
Server → Store session in DB/Memory
Client → Sends session ID cookie
Server → Checks session store
```

Server remembers user → **stateful**

Problems:

* Hard to scale
* Session storage needed
* Load balancer issues

---

## Stateless (Token Based Login / JWT)

Server does NOT store session.

```text
Client → Login
Server → Sends Token
Client → Stores Token
Client → Sends token with every request
Server → Verifies token
```

Server doesn’t remember anything → **stateless**

Better for:

* Scaling
* Microservices
* Mobile apps
* APIs

# 4. Tokens — Concept

Token = proof that user is authenticated.

Client sends token in header:

```http
Authorization: Bearer <token>
```

**Bearer = the one who carries the token**

If you have the token → you are treated as authenticated.

---

# 5. JWT Structure

JWT looks like:

```text
XXXX.YYYY.ZZZZ
Header.Payload.Signature
```

### Header

* algorithm
* token type

### Payload

* user id
* email
* role
* expiry

### Signature

Created using:

```
header + payload + secret
```

Using algorithm like:
**HMAC (Hash-based Message Authentication Code)**

Important:

> Signature ensures token was generated by your server.

---

# 6. JWT Methods

Common methods:

```js
jwt.sign(payload, secret, options)
jwt.verify(token, secret)
jwt.decode(token)
```

| Method | Use                      |
| ------ | ------------------------ |
| sign   | create token             |
| verify | verify token + signature |
| decode | read payload only        |

Important:

> Never trust decode() alone. Always verify().

# 7. Access Token vs Refresh Token

Very important system design concept.

| Access Token            | Refresh Token                |
| ----------------------- | ---------------------------- |
| Short life              | Long life                    |
| Used in API requests    | Used to get new access token |
| 15 min / 30 min         | 7 days / 30 days             |
| Stored in memory/cookie | Stored in DB                 |
| Sent with every request | Sent only for refresh        |

### Flow

```text
Login
   ↓
Server sends:
   Access Token
   Refresh Token
   ↓
Access token expires
   ↓
Client sends Refresh Token
   ↓
Server verifies refresh token
   ↓
Server sends new Access Token
```

This is **token lifecycle + rotation**.

# 8. Hash vs Encryption (CS Fundamentals)

Very important backend concept.

| Hash               | Encryption             |
| ------------------ | ---------------------- |
| One way            | Two way                |
| Cannot decrypt     | Can decrypt            |
| Used for passwords | Used for data transfer |
| bcrypt, sha256     | AES, RSA               |

### Password Storage

Never store password.

Store:

```
hash(password)
```

Then during login:

```
hash(input password) === stored hash
```

# 9. Crypto Functions

You wrote:

```js
crypto.randomBytes()
crypto.createHash()
```

Difference:

| Function    | Use                 |
| ----------- | ------------------- |
| randomBytes | tokens, reset token |
| createHash  | hashing data        |
| bcrypt      | password hashing    |

Example uses:

* Email verification token
* Reset password token
* API keys
* Refresh token
* Session tokens

# 10. Important Mongoose Trick

You wrote:

```js
findOne.select("+password")
```

If schema has:

```js
password: { type: String, select: false }
```

Then password won’t be returned normally.

But during login you need password → so:

```js
User.findOne({ email }).select("+password")
```

Very important real-world pattern.

# 11. Refresh Token Storage

You wrote something important:

```js

```

Difference:

| Value     | Meaning             |
| --------- | ------------------- |
| null      | intentionally empty |
| undefined | not set             |
| ""        | empty string        |

Usually:

```js
user.refreshToken = null
```

during logout.

# 12. Authentication Flow (Full System Flow)

Full production auth flow:

```text
Register
   ↓
Hash password
   ↓
Save user
   ↓
Login
   ↓
Check password hash
   ↓
Generate Access Token
Generate Refresh Token
   ↓
Store Refresh Token in DB
   ↓
Send tokens to client
   ↓
Client sends Access Token in API requests
   ↓
Server verifies Access Token
   ↓
If expired → use Refresh Token
   ↓
Generate new Access Token
```

This is **complete auth system architecture**.
